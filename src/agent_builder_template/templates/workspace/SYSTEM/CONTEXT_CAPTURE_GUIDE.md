# Context Capture Guide

Use this guide when an agent is filling or refining project context.

## Goal

Capture enough context to make the next delivery slice safe and clear without inventing unnecessary detail.

## Working Rules

- Start from `PROJECT/PROJECT_BRIEF.md`.
- Distinguish facts from assumptions.
- Mark unknowns explicitly instead of smoothing them over.
- Prefer context that helps the next decision, task, or approval.
- Escalate architecture, scope, cost, and security questions instead of guessing.

## Recommended Order

1. Clarify intended users or stakeholders.
2. Clarify real constraints and no-go areas.
3. Clarify success measures.
4. Clarify stack direction and architecture boundaries.
5. Clarify approval-sensitive decisions.

## Good Standard

The context is good enough when:

- the next task can be planned without major guessing
- the architecture direction is stable enough for the next slice
- approval-sensitive areas are visible
- the user does not need to restate the same basics in every run
