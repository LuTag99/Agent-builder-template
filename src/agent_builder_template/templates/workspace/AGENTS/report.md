# Report

## Purpose

The Report role turns a run into a concise record that supports decisions, handoffs, and follow-up work. Its job is to summarize what happened without burying the reader in unnecessary detail.

## When To Use It

Use the Report role at the end of a task, iteration, or review cycle. A report is useful whenever someone needs to understand the outcome quickly and decide what should happen next.

## Inputs

The Report role should gather:

- the original goal or request
- the work completed
- validation or review results
- any open risks, issues, or next actions

## Outputs

The Report role should produce a short, decision-useful summary. It should be easy to scan and useful to someone who was not involved in the work directly.

## Reporting Structure

Use this structure unless a project needs a small variation:

### Goal

State the task in one or two sentences.

### Work Completed

List the key actions taken or changes made.

### Result

State the current outcome clearly. If the work is partial, say so.

### Risks Or Open Points

Record any unresolved issues, limitations, or areas of uncertainty.

### Recommended Next Steps

State the most useful next action, if one exists.

## Reporting Guidance

- Keep the report concise.
- Prefer facts and decisions over narrative.
- Separate completed work from open items.
- Do not repeat project context unless it is directly relevant.
- Make it clear whether the task is done, blocked, or awaiting follow-up.
- Point directly to the next useful action when the work is not fully complete.

The Report role is successful when the next reader can understand the outcome quickly and continue the work without searching through the full history.
